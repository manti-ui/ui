# Zag.js coverage

The driving goal of Manti UI is to adapt **every** Zag.js component machine. Zag
ships framework-agnostic state machines that own the hard parts — keyboard
handling, focus management, forms, and ARIA — and Manti wraps each one in a thin
renderer with token-driven CSS. This page tracks the goal, the pattern, and where
coverage stands today.

## The goal

Each adapter keeps the renderer thin, with a strict separation of concerns:

- **`@manti-ui/folds`** — the Zag machine. Owns all behavior and re-exports
  `@zag-js/*` so renderers never import Zag directly.
- **`@manti-ui/styles`** — the CSS, keyed to `data-scope` / `data-part` /
  `data-state`. Owns appearance.
- **`@manti-ui/react`** — the React component. Only wires the two together, with a
  colocated story.

## The adapter pattern

A behavioral component follows one shape — `useMachine` to start the service,
`connect` to get the prop-getter API, then render the anatomy:

```tsx
const service = useMachine(component.machine, { id: useId(), ...props });
const api = component.connect(service, normalizeProps);
return <el {...api.getRootProps()}>{/* parts via api.getXxxProps() */}</el>;
```

The prop getters emit the public `data-scope` / `data-part` / `data-state`
anatomy attributes, which the CSS targets. Collection components (Tabs, Accordion,
RadioGroup, …) currently take a data-driven `items` prop; promoting these to
compound (`Tabs.Trigger`, …) APIs is a planned refinement once breadth is complete.

repository, but it ships in no package and has no documentation page — treat it

## Variants that reuse a batch package

Several components are variants built on a machine already listed above, rather
than new adapters: Context/Nested Menu (`menu`), Range Slider (`slider`),
Segmented Control (`radio-group`), Drawer (`dialog`), Circular vs Linear Progress
(`progress`), and Date Input (`date-picker`).

For the documentation status of each adapted component, see
[Components](/components).
