# Color & variants

Manti UI's palette comes from the mantƒ± kitchen ‚Äî warm, edible references turned
into perceptually-smooth color. Color is split in two: fixed **primitive ramps**
you rarely touch directly, and **variants** ‚Äî the single color-and-emphasis
dimension that components actually read.

## The OKLCH primitive ramps

Six ramps run from `50` to `950`, each authored in **OKLCH** so steps are
perceptually even ‚Äî equal jumps in lightness look equal to the eye, and hues stay
stable across the scale. These values are fixed and identical in light and dark.

| Ramp     | Role    | Reference    |
| -------- | ------- | ------------ |
| `gray`   | neutral | warm neutral |
| `orange` | primary | warm orange  |
| `red`    | danger  | hot red      |
| `green`  | ‚Äî       | fresh green  |
| `amber`  | ‚Äî       | golden amber |
| `blue`   | ‚Äî       | calm blue    |

Each step is a `--manti-{ramp}-{step}` custom property ‚Äî `--manti-orange-500`,
`--manti-gray-950`, and so on.

Per the [design signature](/foundations/design-signature), these ramps never
appear as gradients or a brand accent. They feed the variants below, and the warm
`gray` ramp also backs the cool monochrome surfaces and text.

## The five variants

Variant-driven components ‚Äî Button, Badge, Alert, Switch, and friends ‚Äî never
hard-code a hue. They set a `data-variant` and read a small, uniform vocabulary
of `--variant-*` variables. A single `variant` prop carries **both** the hue and
the emphasis. Manti UI ships five built-in variants:

- `primary` ‚Äî orange solid, the highest-emphasis signature accent
- `secondary` ‚Äî neutral soft, the quiet default
- `tertiary` ‚Äî neutral ghost (text only), the lowest emphasis
- `danger` ‚Äî red solid, the semantic hue
- `outline` ‚Äî neutral bordered: a transparent background with a visible border

The first three form an **emphasis ladder** (`primary` ‚Üí `secondary` ‚Üí
`tertiary`); `danger` is the one semantic hue and `outline` is a neutral bordered
treatment.

## How a variant maps to `--variant-*`

Selecting `[data-variant='primary']` swaps in a complete set of role variables
that every variant-driven component consumes the same way:

```
--variant-solid            --variant-soft-bg          --variant-border
--variant-solid-hover      --variant-soft-bg-hover    --variant-text
--variant-solid-active     --variant-soft-text        --variant-ring
--variant-on-solid
```

Because a component only ever reads `--variant-*` ‚Äî never a raw ramp step ‚Äî the
same markup restyles completely when you change its variant:

```tsx
<Button variant="primary">Save</Button>
<Button variant="danger">Delete</Button>
```

The variant values themselves use `light-dark()` to stay theme-aware and
`color-mix()` to derive soft surfaces and hover states from the resolved solid.

## Custom variants

Any string is a valid variant. To add your own, declare a `[data-variant]` block
that defines the full `--variant-*` vocabulary, then pass it:

```css
[data-variant='brand'] {
  --variant-solid: oklch(0.62 0.16 270);
  --variant-solid-hover: oklch(0.58 0.16 270);
  --variant-solid-active: oklch(0.54 0.16 270);
  --variant-on-solid: var(--manti-text-on-accent);
  --variant-soft-bg: oklch(0.62 0.16 270 / 0.16);
  --variant-soft-bg-hover: oklch(0.62 0.16 270 / 0.24);
  --variant-soft-text: oklch(0.72 0.14 270);
  --variant-border: oklch(0.62 0.16 270 / 0.4);
  --variant-text: oklch(0.72 0.14 270);
  --variant-ring: oklch(0.62 0.16 270);
}
```

```tsx
<Badge variant="brand">New</Badge>
```

The built-in variants keep TypeScript autocomplete via `MantiVariant`, while
still accepting any custom string. Continue to [Design tokens](/foundations/tokens)
for the full token contract.
